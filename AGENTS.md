# Agent 共通ガイドライン

## このファイルについて

このファイルはエージェントの振る舞いを定める。
リポジトリ固有の詳細情報は `docs/` に委ねる。
docs 配下の文書を探す・追加する前に、必ず [`docs/INDEX.md`](./docs/INDEX.md) で入口と置き場を確認する。

エージェントから見て、Git にない知識は存在しないのと同じである。
再利用すべき判断・学んだ事実は、必ず `docs/` またはコードに残すこと。

## 言語

会話・レビュー・コメント・ドキュメントは**日本語**。
専門用語は短く補足する

## あなたの作業の流れ

1. Plan:

- `docs/`などの関連ファイルを確認する

2. Write Test:

- 純粋関数であればunitテスト、API であれば統合テスト、フロント画面の変更であれば統合テストとE2Eテストを必ず実装する
- あまりにも単純な関数は無駄にテストを作成しないで良い。unitテストの実装は重要または複雑なロジックに絞る
- テストは「これがあることで業務上の重要なフローが守れているか？」「人間の手動QAを減らせているか？」という視点でしっかりとレビューすること

3. Implement:

- コード変更は 小さく、レイヤーに収まる単位で 行う。1 PR = 1 論理的変更。
- すでに他に重複実装がないかを確認する
- 関数を細切れに分割しすぎて、処理全体の流れやつながりが追いづらくなっていないか確認する

4. Verify:

- 以下がグリーンであることを確認するまで決して完了としない
  - 静的解析
    - `bun lint`
    - `pnpm format:check`
  - test
    - `pnpm test:unit`
    - `pnpm test:int`
    - `pnpm test:e2e`

5. Document Update：

- 振る舞いやアーキテクチャに影響する変更は docs/ の該当ファイルを 同じPR内で 必ず更新する
- docs 配下に文書を追加・移動するときは [`docs/INDEX.md`](./docs/INDEX.md) で入口と置き場を確認する
- システムの広範囲に関わる設計や重要な仕様の変更をした際、トレードオフを後から確認できるように、実装より先に [`docs/adr/`](./docs/adr/) の ADR を追加または更新する

6. Rule Update:

- 何かミスを犯して修正した場合、それを記録していく。3回以上同じミスを繰り返した場合、重要な遵守事項として同じミスが再発しないような対策を行う。
- 例えば、決定論的に防げるミスはコミット前の lefthook で検証を行う仕組みを追加していく
- 決定論的に防げないものは Agent Rule などを設定する

6. Skill Update:

- 定型的な作業の繰り返しがないか積極的に確認し、skillとして貯めていく
- ただし、1週間以上使われてないものは積極的に削除する

## 何を信頼して正しさを検証するか？(フィードバックセンサー)

### 強く信頼してよいもの: 機械が判定する結果（計算的センサー）

linter、type checker、test の結果は、**決定論的(同じ入力なら同じ結果)**で嘘をつかない。違反があればまず信じて修正する。

- テストが落ちたら、必ず根本原因を修正する。
- テストの無効化、catch で握りつぶす等の暫定回避は禁止。

### 参考程度にとどめるもの: AI が生成した情報（推論的センサー）

AI が書いた PR サマリ、コミットメッセージ、レビューコメントなどは、もっともらしく見えても**確率的で、間違いを含むことがある**。

- 上記の「機械が判定する結果」と矛盾した場合は、必ず機械の方を信じる。
- 重要な判断の前には、要約ではなく元情報(コード、ログ、ファイル)を直接確認する。

### そもそも検証手段が存在しないもの（欠落センサー）

「ユーザーが本当に求めている価値を満たしているか」の最終的な検証は、機械にもAIにもできない。

- タスクの受け入れ条件が曖昧な場合、実装に進む前に Plan で明示的に質問する。
- 情報を「推測」で補うことは禁止。

## コンテキスト衛生

- 新しい知識はリポジトリに残す:
  - 作業中に学んだ非自明な事実(API の癖、ライブラリのバグ、合意された設計判断)は必ず docs/ に書き残す。
- サブエージェントを使ってコンテキストを分離する
  - **使うべき場面**: 大量のファイルを読む探索・調査(結論だけ親に返す)、独立して並列実行できるタスク、レビュー・検証(自己評価バイアスの構造的回避)
  - **使うべきでない場面**: 横断的なリファクタ、設計判断を含む実装(密な文脈共有が必要なため、要約で渡すと情報が落ちる)
