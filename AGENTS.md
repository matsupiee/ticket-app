# Agent 共通ガイドライン

## 基本方針

- 会話・レビュー・コメント・ドキュメントは日本語で記述し、専門用語には短い補足を付ける。
- リポジトリ固有の情報は `docs/` またはコードを正本とする。
- `docs/` を探す・追加・移動する前に、必ず [`docs/INDEX.md`](./docs/INDEX.md) を確認する。
- 再利用すべき設計判断や非自明な知識は、同じPR内でリポジトリに残す。

## 作業フロー

### 1. Plan

- 関連するコードと `docs/` を直接確認する。
- 受け入れ条件が曖昧な場合は、実装前に質問する。
- システム全体に影響する設計・仕様変更は、実装前に [`docs/adr/`](./docs/adr/) のADRを追加または更新する。

### 2. Test Design

変更内容に応じて、業務上重要なフローと手動QAを減らせるテストを実装する。

- 純粋関数: 重要または複雑なロジックにunitテスト
- API: 統合テスト
- フロント画面: 統合テストとE2Eテスト
- 単純な処理には、価値の低いテストを追加しない

### 3. Implement

- 1 PRを1つの論理的変更に収め、変更を小さく保つ。
- 既存の重複実装がないか確認する。
- 関数を細分化しすぎず、処理全体を追える構造にする。

### 4. Verify

以下がすべて成功するまで完了としない。

```sh
bun test:unit
bun test:int
bun test:e2e

bun run dev # apiサーバー、web、web-admin-organizer、web-admin-platform がエラーなく立ち上がることを確認する
```

テスト失敗は根本原因を修正する。テストの無効化や、例外を握りつぶす回避は禁止する。

### 5. Document

振る舞い・仕様・アーキテクチャに影響する変更は、同じPR内で関連ドキュメントを更新する。

### 6. Improve

- 同じミスを3回以上繰り返した場合、再発防止策を追加する。
- 決定論的に検出できる問題は、linter・テスト・lefthookなどで機械的に防ぐ。
- 機械的に検出できない問題はAgent Ruleとして残す。
- 定型作業はskill化を検討し、1週間以上使われていないskillは削除を検討する。

## 7. Self review

全工程が完了したら、最終回答前にレビュー用サブエージェントを1つ起動する。

レビュー用サブエージェントには、現在の差分を読み取り専用で確認させ、以下を優先して指摘させる。

- バグ・仕様逸脱・回帰
- テスト不足
- セキュリティ・権限・データ整合性リスク
- 不要な複雑化や既存方針との不一致
- docs/coding-pattern などで説明されている規約との乖離

メインエージェントは、レビュー結果を確認し、必要な修正を行ってから最終回答する。

## 正しさの検証

優先順位は次のとおり。

1. **コード・ログ・linter・type checker・test**　→ 問題があれば即修正する
2. **AIによる要約・レビュー・説明** → 間違いを含むことがあるので参考程度にとどめる
3. **検証手段のない推測** → 例えば、「ユーザー価値を満たしているか」のような問いを検証することが難しい。推測で補わずに、人間に必ず質問すること。

## コンテキスト管理

非自明な調査結果、APIの仕様、ライブラリの不具合、合意された設計判断は、再利用できる形でコードまたは `docs/` に残す。

サブエージェントは次の場合に使用する。

- 大量ファイルの探索・調査
- 独立して並列実行できる作業
- レビューや第三者視点での検証

横断的なリファクタや、密な文脈共有が必要な設計・実装には使用しない。
