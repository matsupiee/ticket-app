# テスト

- テストファイルは実装ファイルの近くに置く。
  - `packages/api/src/test` は `global-setup.ts` など、テスト全体で使う共通基盤だけを置く。
  - `packages/api/src/test` に `*.unit.test.ts` / `*.integration.test.ts` / `*.test.ts` / `*.spec.ts` を置かない。
  - APIルートのテストは対象APIディレクトリの `handler.integration.test.ts` に置く。
  - `shared` 配下の共通ロジックをテストする場合は、対象ファイルと同じディレクトリに `<対象ファイル名>.unit.test.ts` または `<対象ファイル名>.integration.test.ts` を置く。

- ヘルパー関数は作らない。
  - テストコードではDRYは不要
  - 各自のテストケースブロックを読むだけで何をしているのか読めること、各自のテストケースの独立性を保つことを優先する。
  - DB データ操作についても、`db.**.create({...})` / `db.**.get({...})` とそのまま書いてある方が何が起きているかすぐに確かめやすい。

- シードデータ作成のために各テストケースで使い回す共有オブジェクトを設けることは許容。
  - ただし、どのテストでも結果に影響しないフィールドだけ共有オブジェクトに含めて良い。
  - テストの結果に影響する値は、共有オブジェクトに埋め込まずに各テストケースの中で明示的に書く。

- API 統合テストでは、Prisma Cliant をモック化せず実際の DB を使って検証する。

- API 統合テストでは、最低限、次の観点を確認する。

  - 正常な入力で期待する結果が返る
  - 不正な入力や状態で適切なエラーになる
  - 必要なデータが正しく作成・更新・削除される
  - 権限のないユーザーが処理を実行できない
  - 関連データとの整合性が維持される
